How it works

Every iterator node reads from a single counter — the global_run output of an Iteration Controller. Each time you queue a generation, the counter advances by one. Each iterator uses that counter to decide which file or model to pick, making every run a unique combination.

The key to controlling which dimension cycles fastest is two values that every iterator passes along:

ValueMeaningHow to use it
cycle_every How many global runs before this iterator advances to the next item Leave at 1 for the fastest (innermost) iterator. Wire step_size from a faster iterator here to make this one slower.
step_size (output) cycle_every × files_found — the combined size of this dimension and everything inside it Wire into cycle_every of the next (slower) iterator. The last iterator's step_size = total runs needed.

Example: 3 subject images (fast) × 4 prompts (slow) = 12 total runs.

Installation

  1. Clone or copy the prompt_file_iterator folder into your ComfyUI/custom_nodes/ directory.
  2. Restart ComfyUI. The new nodes appear under the Iterators category in the node menu.
  3. Prepare your content folders — one folder of .txt prompt files, one folder of subject images, etc.
All nodes in this pack are found under Add Node → Iterators. The original Prompt File Iterator node (legacy) remains under Prompt Iterator.

Workflow 1 — Prompts × Checkpoints (txt2img)

Use this to test how different checkpoints handle the same set of prompts, without touching the workflow between runs.

What you need

  • A folder of .txt files — one prompt per file (e.g. D:\prompts\)
  • Two or more checkpoints installed in ComfyUI

Setup steps

  1. Open the workflow. Set the Prompt Iterator directory to your prompts folder using the Browse button.
  2. Open Checkpoint Iterator. Click + Add Checkpoint for each model you want to test and select from the dropdowns.
  3. Adjust KSampler settings (seed, steps, CFG) as desired.
  4. Check the step_size output of the last iterator (Checkpoint Iterator) — this is your total run count. Queue that many times.

Workflow 2 — Images × Prompts (img2img)

Use this to apply a set of creative prompts to a collection of subject images — for example, testing style variations on reference photos.

What you need

  • A folder of subject images (png/jpg/webp etc.) — e.g. D:\subjects\
  • A folder of .txt prompt files — e.g. D:\prompts\
  • One checkpoint loaded statically via a standard CheckpointLoaderSimple node

Setup steps

  1. Set Image Iterator directory to your subjects folder. Set cycle_every = 1 (images are the fast dimension).
  2. Set Prompt Iterator directory to your prompts folder. Wire Image Iterator → step_size into Prompt Iterator → cycle_every.
  3. Set KSampler denoise to 0.6–0.8 for img2img strength.
  4. The Prompt Iterator's step_size = total runs (images × prompts). Queue that many times.

Key difference from Workflow 1

The Image Iterator outputs a real IMAGE tensor — wire it directly into VAE Encode → pixels. The encoded latent feeds into KSampler as the starting point, making this a true img2img workflow.

Dimension order matters. With images as the fast (inner) dimension, runs 0–N cycle through all subjects with prompt 0. Then runs N+1 to 2N cycle through all subjects again with prompt 1. To reverse (all prompts per image), swap which iterator has cycle_every=1.

Workflow 3 — Prompts × LoRAs (txt2img)

Use this to benchmark a set of LoRAs across a fixed prompt set — useful for picking which style LoRA works best with a given subject or concept.

What you need

  • A folder of .txt prompt files
  • Two or more LoRAs installed in ComfyUI (selected via the LoRA Iterator's + Add LoRA button)
  • One checkpoint loaded statically

Setup steps

  1. Set Prompt Iterator directory and leave cycle_every = 1 (prompts are fast).
  2. Open LoRA Iterator. Click + Add LoRA for each LoRA to test.
  3. Wire Prompt Iterator → step_size into LoRA Iterator → cycle_every.
  4. Wire LoRA Iterator → lora_name into LoRA Loader (from String) → lora_name.
  5. Wire the static CheckpointLoaderSimple MODEL and CLIP into LoRA Loader (from String) — its patched model and clip feed the rest of the workflow.
  6. Queue LoRA Iterator → step_size times.
LoRA Loader (from String) works the same as ComfyUI's built-in LoraLoader, but accepts a STRING for the name rather than a dropdown — making it compatible with LoRA Iterator's output.

Checkpoint Loader (from String) & LoRA Loader (from String)

These bridge the gap between the iterator STRING outputs and ComfyUI's MODEL/CLIP/VAE connections. They are functionally identical to ComfyUI's built-in CheckpointLoaderSimple and LoraLoader nodes — the only difference is they accept a STRING input instead of a COMBO dropdown.

Tips & Patterns

How many times should I queue?

The outermost (slowest) iterator's step_size output equals the total number of unique combinations. Connect it to a ShowText or Integer Display node to read the value, then queue that many times. Many third-party extensions also offer a "Queue N times" button.

Adding a third dimension

Chain a third iterator after the second in the same pattern:

  • Iterator A: cycle_every = 1 → step_size = A_count
  • Iterator B: cycle_every ← A's step_size → step_size = A_count × B_count
  • Iterator C: cycle_every ← B's step_size → step_size = A × B × C = total runs

Resetting mid-sequence

Set Iteration Controller → reset = true and queue once. The counter resets to 0 and the first run regenerates combination 0. Flip reset back to false to continue from there.

Using multiple independent workflows

Give each workflow's Iteration Controller a unique session_name. They maintain separate counters even though they write to the same state.json file.

Keeping images organised

Set Save Image → filename_prefix to something like iterator/run_. Use a String Concatenate node wired to file_index outputs to embed the current indices in the filename.
